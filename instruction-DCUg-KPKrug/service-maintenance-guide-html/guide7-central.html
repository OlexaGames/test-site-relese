2. Условия производства работ

2.1. Анализ целостности прикладного программного обеспечения АРМ производится только на находящихся в резерве автоматизированном рабочем месте поездного диспетчера (далее – АРМ ДНЦ), станции связи и шлюзе-сервере стойки поста ДЦ.
2.2. Работа производится электротехническим персоналом, имеющим группу по электробезопасности при работе в электроустановках до 1000 В не ниже III, перед началом работ проинструктированным в установленном порядке.
2.3. К работе допускается обслуживающий персонал, прошедший специальную подготовку и выдержавший испытания в знаниях условий эксплуатации системы в соответствии с процедурами, принятыми в ОАО «РЖД».
2.4. Допуск на объект, оформление записей в журналах, контроль качества работ, выполняемых подрядными организациями, участие в приемке выполненных работ осуществляется эксплуатационным персоналом в соответствии с требованиями Инструкции по техническому обслуживанию и ремонту устройств и систем сигнализации, централизации и блокировки, утвержденной распоряжением ОАО «РЖД» от 30 декабря 2015 г. № 3168р.

5. Обеспечение безопасности движения поездов

5.1. Анализ целостности прикладного программного обеспечения АРМ производится в технологическое «окно» с согласия поездного диспетчера (далее – ДНЦ).
5.2. Работа выполняется без оформления записи в Журнале осмотра путей, стрелочных переводов, устройств сигнализации, централизации и блокировки, связи и контактной сети формы ДУ-46 (далее – Журнал осмотра).
5.3. При выполнении работы обеспечить безопасность движения в соответствии с требованиями Инструкции по обеспечению безопасности движения поездов при технической эксплуатации устройств и систем СЦБ ЦШ-530-11, утвержденной распоряжением ОАО «РЖД» от 20 сентября 2011 г. № 2055р.

7. Технология выполнения работ

Анализ целостности прикладного программного обеспечения АРМ

Анализ целостности файлов программного обеспечения выполняется с целью проверки соответствия файлов эталонным копиям. Проверка выполняется путем сравнения хэш-кодов файлов, рассчитанных по стандартному алгоритму хеширования MD5 для актуальных файлов на момент проверки со значениями эталонного хэш-кодов.

ВНИМАНИЕ: Расчет выполняется с помощью консольной утилиты md5.exe из состава ПО ДЦ ЮГ.
ПРИМЕЧАНИЕ: Утилита принимает два параметра: базовый каталог и шаблон для выборки файлов. Например, запуск команды «md5 c:/armdnc *.exe» выполнит поиск всех файлов с расширением «.exe», рассчитает для них хэш MD5 и выведет на консоль.

7.1. На АРМ ДНЦ, находящемся в резерве, запустить предустановленный пакетный файл «getmd5.bat». При запуске файла «getmd5.bat» в консоли отображается информация, представленная на рисунке 1.

Рисунок 1. Запуск пакетного файла «getmd5.bat».

ПРИМЕЧАНИЕ: Пакетный файл содержит строки запуска утилиты «md5» с различными параметрами для разных типов файлов, а именно:
- «.exe» – исполняемые файлы программного обеспечения.
- «.ini» – конфигурационные файлы программного обеспечения.
- «.mdb» – файлы базы данных информационного обеспечения.
- «.shp» – файлы графических схем.

7.2. Результаты расчета выводятся в текстовый файл «md5.txt» (Рисунки 2а, 2б, 2в, 2г).

Рисунок 2а. Исполняемые файлы программного обеспечения.

Рисунок 2б. Конфигурационные файлы программного обеспечения.

Рисунок 2в. Файлы базы данных информационного обеспечения.

Рисунок 2г. Файлы графических схем.

ВНИМАНИЕ: Текстовый файл «md5.txt» содержит список файлов программного обеспечения и соответствующие хэш-коды.

7.3. Выполнить сравнение результата с эталонными значениями.

ВНИМАНИЕ: Эталонная распечатка (листинг) данного файла хранится вместе с дистрибутивом программного обеспечения.
ВНИМАНИЕ: При выполнении обновления технологического программного обеспечения выполняется контрольный расчет для обновленных файлов с последующей заменой эталонной распечатки (листинга).
ПРИМЕЧАНИЕ: В случае необходимости проведения оценки целостности файлов программного обеспечения выполнить следующие действия: провести расчет хэш-кодов актуальных файлов, а затем сравнить результаты с эталонными значениями.

7.4. В случае отсутствия различий при сравнении результата с эталонными значениями перейти к пункту 7.5. В случае выявления различий при сравнении результата с эталонными значениями передать информацию диспетчеру центра диагностики и мониторинга для организации оповещения разработчика системы (центра сервисного обслуживания) для принятия мер.
7.5. Оформить запись в Журнале осмотра.
Пример записи:
В технологическое «окно» будет произведено переключение с основного АРМ ДНЦ на резервный и управление с резервного АРМ ДНЦ.
ШНД
ДНЦ

7.6. Провести переключение активности с основного АРМ ДНЦ на резервный согласно ТНК ЦШ 0197-2016.
7.7. Выполнить проверку на основном АРМ ДНЦ в соответствии с пунктами 7.1-7.3.
7.8. Выполнить проверку на АРМ ШН в соответствии с пунктами 7.1-7.3.
7.9. Выполнить проверку на основной и резервной станции связи стойки поста ДЦ в соответствии с пунктами 7.1-7.3.
7.10. Выполнить проверку на основном и резервном шлюзе-сервере стойки поста ДЦ в соответствии с пунктами 7.1-7.3.

8. Заключительные мероприятия, оформление результатов работы

После окончания анализа целостности прикладного программного обеспечения АРМ оповестить ДНЦ об окончании работы, сделать запись в журнале ШУ-2 о проведенной работе и внести результаты выполнения работы в систему ЕК АСУИ в соответствии с требованиями, изложенными в разделе 4 «Порядка планирования, учета и контроля выполнения работ в хозяйстве автоматики и телемеханики», утверждённого распоряжением ОАО «РЖД» от 13 января 2020 г. №20/р.
Результаты работы по сопровождению программного обеспечения специализированная организация оформляет в журнале учета выполненных работ по техническому обслуживанию и ремонту устройств и систем ЖАТ по форме, представленной в регламенте взаимодействия между участниками процесса технического обслуживания и ремонта систем и устройств железнодорожной автоматики и телемеханики ОАО «РЖД», осуществляемых специализированными организациями, утвержденном распоряжением ОАО «РЖД» от 30 декабря 2017 г. № 2827р.
